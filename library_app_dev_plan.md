## 0. 한 줄 요약

**순수 HTML/CSS/JavaScript로 만드는 웹앱(PWA)** 으로, PRD의 학생용 기능 7개 전부와 간단한 관리자 화면까지 2일 안에 완성합니다. 설치 프로그램도, 서버도, 앱스토어 심사도 필요 없습니다. 브라우저에서 열고 "홈 화면에 추가"를 누르면 진짜 앱처럼 보입니다.

---

## 1. 왜 이 기술을 골랐나

PRD 원문에는 "iOS, Android 하이브리드 앱"이라고 적혀 있습니다. 하지만 **2일이라는 기간 안에서는 이 요구사항을 그대로 따르면 아무것도 완성하지 못합니다.** 네이티브 앱은 개발 환경 설치(Android Studio, Xcode)에만 반나절이 걸리고, Apple 개발자 계정은 연 $99이며, 스토어 심사는 1~2주가 걸립니다. 2일이라는 예산으로는 "심사 대기 중인 반쪽짜리 앱"이 나옵니다.

그래서 **PWA(Progressive Web App)** 를 선택했습니다. PWA는 웹페이지지만 홈 화면에 아이콘으로 추가할 수 있고, 전체 화면으로 뜨고, 오프라인에서도 동작합니다. 학생 입장에서는 앱과 구별이 거의 안 됩니다.

이 선택이 PRD 요구사항을 오히려 더 잘 만족시키는 부분도 있습니다.

| PRD 요구사항 | PWA로 해결되는 방식 |
|:---|:---|
| iOS + Android 양쪽 지원 | 브라우저만 있으면 되므로 한 번 만들어 양쪽 다 동작 |
| 반응형 UI (태블릿 + 폰) | 웹의 기본 특기. CSS 한 벌로 처리 |
| 로컬 오프라인 모드 (§3-4) | Service Worker + localStorage로 구현. 네이티브보다 오히려 쉬움 |
| TTS 텍스트 읽어주기 (§3-3) | 브라우저 내장 Web Speech API. **코드 5줄이면 끝** |

**단 하나 포기하는 것:** 앱스토어에 정식 등록되지 않습니다. 대신 학생들에게 QR코드나 링크를 나눠주고 "홈 화면에 추가"를 안내하면 됩니다. 학교 환경에서는 오히려 이 방식이 배포가 더 빠릅니다.

### 2.2 핵심 전략: 화면 4종류만 만들면 7개 기능이 전부 된다

초보자가 2일 안에 이만큼을 끝낼 수 있는 이유는, 기능 7개를 각각 만들지 않고 **재사용 가능한 화면 엔진 4개**로 접기 때문입니다.

```
① 가이드 뷰어 (스텝 카드를 순서대로 넘김)
   └─ F-01 도서관 소개    ┐
   └─ F-03 대출/반납      ├─ 코드는 하나, 데이터만 3벌
   └─ F-05 보물찾기       ┘

② 플래시카드 (터치하면 뒤집힘)
   └─ F-02 도서관 사전

③ 스와이프 카드 (좌우로 밀어서 분류)
   └─ F-04 도서관 약속

④ 퀴즈 (문제 → 채점 → 리워드)
   └─ F-06 도전! 도서관 퀴즈

+ 대시보드, 관리자 화면
```

**이것이 이 계획서에서 가장 중요한 문장입니다.** F-01을 만들 때 "도서관 소개 화면"을 만든다고 생각하지 말고, **"스텝 카드를 넘겨보는 화면"** 을 만든다고 생각하세요. 그러면 F-03과 F-05는 `data.js`에 내용을 추가하는 것만으로 공짜로 따라옵니다. 1.5시간 걸릴 작업이 3개가 아니라 1개가 됩니다.

### 2.3 관리자 화면에 대한 정직한 한계

관리자 화면을 포함하기로 했지만, **서버 없이 만들기 때문에 실제 운영에는 쓸 수 없다는 점을 분명히 알고 시작해야 합니다.**

- 데이터가 **각 학생의 폰 안에만** 저장됩니다.
- 따라서 사서 선생님이 교무실 PC에서 관리자 화면을 열어도 **학생 데이터가 하나도 안 보입니다.** (그 PC의 데이터만 보임)
- 학생이 브라우저 기록을 지우면 데이터가 사라집니다.

**그래서 이렇게 씁니다:** 관리자 화면은 "학생이 자기 폰을 사서 선생님께 보여주면, 선생님이 PIN을 입력해서 쿠폰을 '사용 완료' 처리하는 화면"으로 만듭니다. 즉 **오프라인 쿠폰 검수 도구**입니다. 이건 서버 없이도 실제로 유용합니다.

전교생 현황 통계가 진짜로 필요하다면, 2일 이후에 Firebase나 Google Sheets 연동을 붙이는 3일차 작업이 됩니다. 이번 범위는 아닙니다.

### 2.4 PRD에서 의도적으로 축소하는 것

버리는 게 아니라 **2일에 맞게 대체하는 것**입니다. 각각 이유가 있습니다.

| PRD 원안 | 이번 구현 | 이유 |
|:---|:---|:---|
| 귀여운 2D 일러스트 평면도 (F-01) | CSS 그리드로 그린 도식 배치도 | 일러스트 제작은 디자이너 영역. 2일 안에 그릴 수 없음 |
| 캐릭터 일러스트 (도서관 요정 등) | 이모지 (📚🧚🐕) | 위와 동일. 이모지도 저학년에게 충분히 친근함 |
| 사서 선생님 사진 | 이모지 + 환영 인사 텍스트 | 사진 파일이 준비되면 나중에 1줄로 교체 가능하게 설계 |
| 실시간 공지사항 | `data.js`에 하드코딩 | "실시간"은 서버가 필요. 선생님이 파일 한 줄 고치는 방식으로 대체 |
| 학생증 바코드 (실제 학사 시스템 연동) | 학번 입력 → JsBarcode로 바코드 생성 | 학교 시스템 연동은 협의·인증이 필요한 별도 프로젝트 |
| 앱스토어 배포 | GitHub Pages + "홈 화면에 추가" | 1절 참조 |

**중요:** 위 항목들은 전부 나중에 교체할 수 있도록 설계합니다. 이모지 자리에 이미지 파일 경로만 넣으면 되게 만들어 두세요.

## 3. 파일 구조

폴더 하나에 파일 8개. 이게 전부입니다.

```
c:\project\library\
├── index.html          ← 화면 뼈대 (모든 화면이 여기 안에 있고 JS가 보여주고 숨김)
├── style.css           ← 디자인 전부
├── app.js              ← 화면 전환, 이벤트 처리
├── data.js             ← 모든 콘텐츠 (단어, 퀴즈, 가이드 스텝) ⭐
├── storage.js          ← 저장/불러오기 (진행률, 배지, 쿠폰)
├── barcode.js          ← 바코드 그리기 (직접 구현, 외부 라이브러리 없음)
├── manifest.json       ← "홈 화면에 추가"용 설정
├── sw.js               ← 오프라인 캐싱
└── icons/
    ├── icon-192.png
    └── icon-512.png
```

> **계획 변경 1건:** 원래는 바코드용으로 JsBarcode 라이브러리를 다운로드할 예정이었으나,
> CODE 39 방식을 직접 구현했습니다(`barcode.js`, 40줄). 외부 파일 의존이 사라져서
> 오프라인 동작이 더 확실해지고, `sw.js`의 캐시 목록도 짧아집니다.

**`data.js`를 코드와 분리하는 이유:** 나중에 사서 선생님이 퀴즈 문제나 도서관 이용 시간을 바꾸고 싶을 때, `data.js` 파일 하나만 열어서 글자를 고치면 됩니다. HTML이나 JS를 건드릴 필요가 없습니다. 초보자에게도 유리한데, 콘텐츠 입력 작업과 코딩 작업이 섞이지 않아서 머리가 덜 아픕니다.

## 4. 데이터 설계

### 4.1 저장 구조 (localStorage)

키 하나에 객체 하나. 통째로 저장하고 통째로 읽습니다. 초보자에게는 이게 가장 안전합니다.

```javascript
// localStorage key: "library_explorer_v1"
{
  student: {
    name: "김도서",
    grade: 3,
    studentId: "30115"        // 바코드로 변환할 값
  },
  progress: {                 // 각 단계 완료 여부
    f01: false,               // 도서관 소개
    f02: false,               // 도서관 사전
    f03: false,               // 빌리고 반납하기
    f04: false,               // 도서관 약속
    f05: false                // 보물찾기
  },
  quiz: {
    passed: false,
    bestScore: 0,
    attemptCount: 0
  },
  badges: [],                 // 예: ["explorer"]
  coupons: [                  // 퀴즈 통과 시 발급
    // { id, type, code, issuedAt, usedAt }
  ],
  loans: [                    // D-Day 계산용 (학생이 직접 입력)
    // { title: "마당을 나온 암탉", dueDate: "2026-07-22" }
  ]
}
```

**진행률 계산:** `progress` 안에서 `true`인 개수 ÷ 5 × 100. 대시보드 진행률 바에 그대로 씁니다.

## 6. 구현 시 알아둘 것

### 6.1 화면 전환은 이렇게 (라우터가 필요 없는 이유)

라이브러리 없이, 이 개념 하나로 전부 처리됩니다.

```javascript
// 모든 화면을 HTML에 다 넣어두고, 하나만 보여주고 나머지는 숨긴다.
function showScreen(id) {
  document.querySelectorAll('.screen').forEach(el => el.hidden = true);
  document.getElementById(id).hidden = false;
}
```

React가 하는 일의 90%가 이겁니다. 이 앱 규모에서는 이 4줄로 충분합니다.

### 6.2 TTS는 정말로 5줄

PRD §3-3의 "TTS 옵션 탑재 권장"은 이걸로 끝납니다.

```javascript
function speak(text) {
  const u = new SpeechSynthesisUtterance(text);
  u.lang = 'ko-KR';
  u.rate = 0.9;              // 저학년을 위해 조금 느리게
  speechSynthesis.speak(u);
}
```

각 카드에 🔊 버튼을 달고 `speak(카드내용)`을 호출하세요. 저학년 배려라는 PRD의 목표 대비 투입 시간이 압도적으로 적은, 이 앱 최고의 가성비 기능입니다.

### 6.3 오프라인은 "파일 목록 적기"가 전부

```javascript
// sw.js — 캐싱할 파일 목록만 적으면 됨
const FILES = ['./', './index.html', './style.css', './app.js', './data.js', ...];
```

PRD §3-4의 "최초 설치 시 캐싱 저장" 요구사항이 이 목록 하나로 충족됩니다. **파일을 추가하면 이 목록에도 추가하는 걸 잊지 마세요.** 초보자가 가장 자주 겪는 함정입니다 — 파일을 추가했는데 오프라인에서만 안 되는 증상이 나옵니다.

### 6.4 오프라인 캐시 때문에 수정이 반영 안 될 때

Service Worker를 켜면 **코드를 고쳐도 브라우저가 옛날 버전을 계속 보여줍니다.** 초보자가 여기서 "내 코드가 왜 안 먹지?" 하며 1시간을 날리는 일이 아주 흔합니다.

**그래서 Day 2의 PWA 작업을 일부러 맨 마지막 근처에 배치했습니다.** 개발 중에는 Service Worker를 끄고, 다 만든 뒤에 켜세요. 개발 중에 확인이 필요하면 개발자도구 → Application → "Update on reload"를 체크하면 됩니다.

### 6.5 막혔을 때의 순서

1. **개발자도구 콘솔(F12)을 먼저 본다.** 빨간 글씨가 답을 알려줍니다. 초보자의 90%는 이걸 안 봐서 헤맵니다.
2. `console.log()`로 값을 찍어본다.
3. 그래도 모르면 에러 메시지를 그대로 복사해서 질문한다.
